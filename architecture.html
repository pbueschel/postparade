Internal architecture brief

The matching layer for the racing industry.

PostParade is a two-sided SaaS: trainers find the right race for each horse, and racing offices fill their cards — one recommendation engine scoring both directions. Architecturally we sit beside the incumbents rather than replacing them: Equibase stays the data of record (charts, past performances, codes) and InCompass stays the racing-office system of record. PostParade is the placement and matching layer stitched across them.

How it's staged

One engine, three stages of infrastructure.

The engine is written once and travels forward unchanged; only the plumbing around it grows. We add a backend precisely when two real parties need to share state — not before.

★ Per-track config registry — the jurisdictional-variance store
Entry-close hours · preference system · scratch times · state funds · Lasix rules · coupled-entry policy — configurable per track, never hard-coded. This registry is what lets one codebase serve every jurisdiction.
Tenancy & rollout

The track is the anchor tenant.

A track org is the tenant we sell to and the unit we roll out. Activating a meet lights up the marketplace for that circuit's trainers, and the per-track config registry absorbs the jurisdictional differences that would otherwise fork the codebase. That registry is the enterprise moat.

AnchorTrack org = tenant

Each track (and its racing office) is a tenant. Stable orgs attach to the circuits they run on. Every row is org-scoped, so tenancy is enforced at the data layer, not just the UI.

TriggerA meet activates the market

Publishing a meet opens the two-sided marketplace for that circuit's trainers — races become submittable, and the track can request eligible horses. No meet, no marketplace.

MoatConfig registry = variance

Entry-close hours, preference systems, scratch times, state funds and medication rules all live in the per-track registry. Onboarding a new jurisdiction is configuration, not engineering.

Why tracks first: rolling out per-track means each go-live is contained, the config registry captures that jurisdiction's rules once, and the meet activation pulls its trainers onto the platform behind it. The rollout unit and the sales unit are the same object.
Roadmap, not fantasy

Build-out sequencing.

Every box in the enterprise diagram maps to a stage in the plan, so the picture reads as a roadmap. Much of the engine already exists in the demo; the rest is staged behind real customer need.

Architecture box First appears What exists / what's added
Matching / recommendation engine Stage 1 Eligibility gates + fit + ship + acceptance already implemented client-side; v1 adds vet/medication gates and draw-in probability.
Trainer & track web apps Stage 1 Both workspaces run today on static Pages with a seed data module and localStorage submit/request.
Race/Card, Stable, Entry, Incentive services Stage 2 Extracted from the seed module into a thin Bun + Hono API over bun:sqlite when two roles must share state.
Magic-link auth (single-tenant) Stage 2 Two-role login; org-scoped AuthN/AuthZ generalizes it in Stage 3.
Notification service Stage 2 Email first (requests + close times); SMS/push via Twilio in Stage 3.
Postgres + replicas, object store, cache Stage 3 Multi-tenant data layer replaces the single SQLite file when scale and tenancy demand it.
API gateway, org-scoped roles, rate limiting Stage 3 Edge tier introduced with true multi-tenancy (stable orgs vs track orgs).
Equibase adapter · condition-book parser Stage 3 Parser prototyped early as a shared playground; productionized as a pipeline with a human QA queue when the Equibase feed lands.
InCompass interop · HISA / state feeds Stage 3 Interop layer once tracks transact on-platform; per-state rules flow into the config registry.
Control plane (billing, rollout, config registry) Stage 3 Stood up as we onboard paying track orgs; Stripe billing, per-track feature flags, jurisdictional config store.
Read this way: the demo is not a throwaway. Its engine and data shape are the Stage 3 matching service and API contract — later stages swap the plumbing beneath them, not the core.
